## Context

Сейчас весь GTK4/Vala-код фактически остался в одном файле `src/statewidget.vala`: там одновременно определены enum состояний, переиспользуемый `StatusStateWidget`, `MainWindow`, `MyApp` и функция `main`. При этом `src/meson.build` все еще ожидает файл `main.vala`, поэтому после переименования исходника структура проекта стала неявной и неудобной для дальнейшего развития.

Пользователь хочет использовать `statewidget.vala` как модуль в собственной реализации `main.vala`. Для этого нужно не только физически разнести код по файлам, но и сделать границу между модулем виджета и кодом приложения явной, чтобы виджет можно было подключать без побочных зависимостей на конкретное окно или точку входа.

## Goals / Non-Goals

**Goals:**
- Выделить `StatusState` и `StatusStateWidget` в самостоятельный исходный модуль.
- Вернуть отдельный `main.vala`, который содержит приложение и использует модульный виджет через публичный API.
- Обновить Meson-конфигурацию так, чтобы сборка учитывала оба файла и успешно компилировалась.
- Сохранить текущее пользовательское поведение виджета и существующую демонстрационную интеграцию в окне.

**Non-Goals:**
- Не менять набор поддерживаемых состояний виджета.
- Не добавлять библиотечную упаковку, namespace-переезд или отдельный shared library target.
- Не перестраивать весь UI-поток приложения сверх необходимого для разделения по файлам.

## Decisions

1. **Оставить `statewidget.vala` только для переиспользуемых сущностей виджета.**  
   В этот файл войдут `StatusState` и `StatusStateWidget`, но не `MainWindow`, `MyApp` и `main`. Это даст чистую модульную границу. Альтернатива — оставить классы окна рядом с виджетом и просто добавить новый `main.vala` — отклонена, потому что тогда модуль все равно останется связанным с приложением.

2. **Создать отдельный `main.vala` с демонстрационным окном, использующим модуль.**  
   Это обеспечит рабочий пример интеграции и одновременно вернет ожидаемую структуру проекта для Meson. Альтернатива с переименованием `statewidget.vala` обратно в `main.vala` не решает исходную задачу модульного переиспользования.

3. **Собрать приложение из двух Vala-файлов в одном executable target.**  
   Для текущего масштаба проекта достаточно перечислить оба исходника в `src/meson.build`. Альтернатива с отдельной библиотечной целью преждевременна и добавляет лишнюю сложность без прямой пользы для текущего запроса.

4. **Сохранить существующий API виджета через конструктор по умолчанию и `set_state`.**  
   Это минимизирует изменения в коде окна и оставляет понятную точку интеграции для пользовательского `main.vala`. Альтернатива с внедрением более сложного контроллера состояния сейчас не нужна.

## Risks / Trade-offs

- **[Risk]** После разделения по файлам можно случайно потерять часть логики демонстрационного окна. → **Mitigation:** Переносить только UI-обвязку в `main.vala`, не изменяя публичный API `StatusStateWidget`.
- **[Risk]** Сборка может не пройти из-за несогласованности списка исходников в Meson. → **Mitigation:** Явно обновить `src/meson.build` и проверить, что target перечисляет оба файла.
- **[Trade-off]** Виджет остается модулем на уровне исходного файла, а не отдельной библиотекой. → **Mitigation:** Сохранить компактный API и структуру, которую позже можно будет вынести в библиотеку без изменения поведения.

## Migration Plan

1. Переместить определения `StatusState` и `StatusStateWidget` в модульную часть `src/statewidget.vala`.
2. Создать `src/main.vala` с `MainWindow`, `MyApp` и `main`, подключающими модуль виджета.
3. Обновить `src/meson.build`, чтобы включить оба исходных файла в сборку executable.
4. Собрать проект и проверить, что виджет по-прежнему работает в окне после разделения.
