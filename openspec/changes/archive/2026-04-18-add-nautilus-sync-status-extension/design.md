## Context

Текущий репозиторий содержит GTK4/Vala frontend, но не имеет интеграции с GNOME Files/Nautilus. При этом backend в `../discohack-daemon` уже публикует D-Bus API `ru.literallycats.daemon` с достаточной информацией для shell-интеграции: свойство `MountPoint`, агрегированные свойства `SyncSummary`/`SyncItems`, методы `GetSyncStatus(path)` и `ListDirectoryStatuses(path)`, а также стандартные `PropertiesChanged`-уведомления при изменении глобального sync-state.

Референс `../desktop/shell_integration/nautilus/syncstate.py` показывает типичный паттерн расширения Nautilus: процесс расширения живет внутри Nautilus, хранит собственный кэш `FileInfo`, получает push-обновления и вызывает `invalidate_extension_info()` для перерисовки эмблем. В нашем случае вместо собственного socket-протокола Nextcloud нужно использовать session D-Bus и уже существующий daemon API.

Главные ограничения:
- расширение не должно подвешивать Nautilus долгими или лишними запросами на каждый файл;
- overlay-эмблемы должны появляться только для путей внутри mount point Яндекс Диска;
- недоступность daemon не должна ломать файловый менеджер;
- проект пока не содержит готового packaging для Nautilus extension и, вероятно, потребует Python runtime alongside существующему Vala frontend.

## Goals / Non-Goals

**Goals:**
- Добавить рабочее Nautilus extension, показывающее эмблемы sync-статуса для файлов и папок внутри mount point Яндекс Диска.
- Использовать существующий D-Bus контракт daemon без добавления нового transport layer.
- Минимизировать количество D-Bus вызовов за счет кэша и батчевого чтения статусов каталога.
- Обеспечить live-refresh overlay-статусов через `PropertiesChanged` и инвалидацию уже наблюдаемых элементов Nautilus.
- Добавить установку расширения и связанных ресурсов в проект так, чтобы интеграцию можно было поставить вместе с приложением.

**Non-Goals:**
- Не добавлять контекстные меню, share actions, публичные ссылки и прочую shell-функциональность по образцу Nextcloud.
- Не менять D-Bus контракт daemon, если текущих `MountPoint`/`GetSyncStatus`/`ListDirectoryStatuses` достаточно.
- Не поддерживать в первом изменении Nemo, Caja и другие файловые менеджеры.
- Не строить отдельный синхронизационный демон или отдельный watcher поверх файловой системы — источником истины остается существующий daemon.

## Decisions

1. **Реализовать расширение на Python через `nautilus-python`, а не как нативный Vala/C extension.**  
   **Почему:** у нас уже есть рабочий референс на Python, для overlay-логики не нужен тяжелый native слой, а интеграция с GObject/GIO D-Bus API в PyGObject естественно сочетается с Nautilus. Это минимизирует объем glue-кода и упрощает итерацию.  
   **Альтернативы:**
   - Нативное расширение на C/Vala с `libnautilus-extension` отклонено как более дорогое по времени и сопровождению.
   - Отдельный helper-процесс отклонен, потому что daemon уже существует и публикует нужный API.

2. **Держать код shell-интеграции в отдельном каталоге и устанавливать его как data-файл Meson, а не смешивать с Vala sources.**  
   **Почему:** Python extension не компилируется вместе с Vala executable и имеет другой runtime/lifecycle. Отдельная папка (например, `nautilus/`) и отдельный `meson.build` дадут чистую границу между приложением и расширением.  
   **Альтернативы:** хранить `.py` внутри `src/` проще краткосрочно, но ухудшает читаемость и сборочную структуру.

3. **Использовать `Gio.DBusProxy`/GIO D-Bus API напрямую к `ru.literallycats.daemon`, а не `dbus-python` или внешние библиотеки.**  
   **Почему:** PyGObject уже нужен для Nautilus extension, а GIO дает стандартную интеграцию с GLib main loop, сигналами `PropertiesChanged` и синхронными/асинхронными вызовами без новой зависимости.  
   **Альтернативы:** `dbus-python` добавляет еще один runtime dependency и хуже вписывается в GObject-экосистему Nautilus.

4. **Определять принадлежность файла к Яндекс Диску через `MountPoint` и локальное преобразование пути `file:// -> abs path -> daemon path`.**  
   Локальный путь Nautilus будет канонизироваться через decoding URI и `realpath`. Если путь не находится под `MountPoint`, extension ничего не делает. Если путь совпадает с mount root, он мапится в `disk:/`; дочерние пути превращаются в `disk:/relative/path`.  
   **Почему:** daemon уже экспортирует локальную точку монтирования, значит extension не должен угадывать ее из конфигов или хардкода.  
   **Альтернативы:** запрос статуса по локальному абсолютному пути потребовал бы менять backend API; для этого изменения это лишнее.

5. **Использовать двухуровневое получение статусов: батч по каталогу + точечный fallback.**  
   - Для элементов при обзоре каталога extension сначала пытается загрузить статусы прямых детей через `ListDirectoryStatuses(parent_daemon_path)` и разложить их в локальный кэш.  
   - Если элемент не попал в батч, если это mount root, или если нужен единичный refresh, используется `GetSyncStatus(path)`.  
   **Почему:** Nautilus вызывает `update_file_info()` много раз; выполнять отдельный D-Bus round-trip на каждый файл слишком дорого. Батч по каталогу повторяет идею референса Nextcloud, но поверх D-Bus методов daemon.  
   **Альтернативы:** всегда вызывать только `GetSyncStatus` проще, но хуже масштабируется на большие каталоги.

6. **Сопоставлять backend-состояния в компактный набор overlay-эмблем, а не рисовать отдельную визуализацию для каждого переходного подтипа.**  
   Предлагаемая маппинг-модель для первой версии:
   - `synced` → success/ok emblem
   - `queued`, `uploading`, `downloading` → syncing emblem
   - `conflict` → conflict/warning emblem
   - `error` → error emblem
   - `unknown` или daemon unavailable → без эмблемы

   **Почему:** Nautilus overlay должен быть мгновенно считываемым и не перегруженным, а переходные backend-состояния можно безопасно свернуть в один визуальный статус «синхронизация идет».  
   **Альтернативы:** отдельные иконки для upload/download/queued дают больше деталей, но потребуют больше ресурсов и сложнее в поддержке.

7. **Обновлять overlay через отслеживание `PropertiesChanged` и сравнение старого/нового множества `SyncItems`.**  
   Расширение будет подписываться на изменения свойств daemon и хранить последнее множество путей из `SyncItems`. Когда свойства меняются, extension вычисляет добавленные/удаленные/измененные пути, конвертирует их в локальные абсолютные пути и вызывает `invalidate_extension_info()` для уже наблюдаемых `FileInfo`, а также для родительских каталогов, чьи children cache могли устареть.  
   **Почему:** daemon не шлет per-path custom signals, но стандартные property updates уже достаточно информативны для инвалидации локального кэша и lazy refetch.  
   **Альтернативы:** периодический polling проще, но создает лишнюю нагрузку и задержку отображения.

8. **Graceful degradation: при ошибках D-Bus extension возвращает «нет эмблемы» и постепенно переподключается, а не показывает ложные состояния.**  
   **Почему:** shell-интеграция не должна ухудшать стабильность Nautilus. Отсутствие overlay лучше, чем зависание, traceback или stale-state, который выглядит как корректный.  
   **Альтернативы:** кэшировать статусы бесконечно опасно — пользователь может видеть неактуальный sync state после рестарта daemon.

9. **Сделать установку расширения и эмблем явной частью Meson-конфигурации.**  
   Понадобится установить Python-файл расширения в каталог `nautilus-python/extensions` и, если будут выбраны собственные overlay icons, установить их в hicolor icon theme. Желательно предусмотреть Meson option/variable для override install path, чтобы локальная разработка и packaging не зависели от одного хардкодного пути.  
   **Почему:** без этого код останется только исходником в репозитории и не станет рабочей shell-интеграцией.  
   **Альтернативы:** ручное копирование файлов в системные каталоги неудобно и плохо воспроизводимо.

## Risks / Trade-offs

- **[Risk]** Nautilus часто вызывает `update_file_info()`, и наивная реализация может породить слишком много D-Bus round-trip'ов. → **Mitigation:** directory-level batching, локальный cache, lazy fallback только для miss-paths.
- **[Risk]** `SyncItems` содержит только bounded active/conflict/error set, поэтому переход в `synced` может выражаться исчезновением пути из списка. → **Mitigation:** сравнивать старое и новое множество путей и инвалидировать также удаленные из `SyncItems` элементы, чтобы они были перечитаны через `GetSyncStatus`.
- **[Risk]** Overlay icon names могут не существовать в текущей теме GNOME. → **Mitigation:** либо выбрать гарантированно доступные emblem names, либо поставить собственный небольшой набор SVG-иконок вместе с проектом.
- **[Risk]** Путь Nautilus может не совпасть с daemon path из-за symlink/canonicalization нюансов. → **Mitigation:** нормализовать оба пути через absolute/realpath и четко определить одно преобразование `MountPoint <-> disk:/`.
- **[Risk]** D-Bus сервис может быть временно недоступен во время старта Nautilus или рестарта daemon. → **Mitigation:** обрабатывать отсутствие bus owner как нормальное состояние, очищать кэш и переподключаться через GLib timeout или lazy reconnect.
- **[Trade-off]** Python extension проще внедрить, но он добавляет еще один runtime stack рядом с Vala frontend. → **Mitigation:** изолировать shell-интеграцию по каталогам и не смешивать ее с основным UI-кодом.

## Migration Plan

1. Добавить каталог shell-интеграции с Python Nautilus extension, D-Bus клиентом и маппингом статусов в эмблемы.
2. Подключить новый каталог к Meson и настроить установку extension-файла (и при необходимости icon assets).
3. Реализовать path mapping `MountPoint <-> disk:/`, directory batching и fallback `GetSyncStatus`.
4. Реализовать подписку на `PropertiesChanged`, кэш наблюдаемых элементов и инвалидацию overlay при обновлениях.
5. Протестировать вручную сценарии: daemon доступен, daemon недоступен, файл внутри mount, файл вне mount, активная синхронизация, конфликт, ошибка, возврат в synced.
6. При необходимости добавить документацию по установке/перезапуску Nautilus после установки extension.

Rollback простой: отключить установку нового Nautilus extension или удалить установленный extension-файл; существующий Vala frontend и daemon контракт останутся без изменений.

## Open Questions

- Хотим ли мы в первой версии поставить собственные branded/status SVG emblem icons или достаточно набора системных эмблем?
- Нужен ли extension принудительный вызов `RequestRefresh(path)` при открытии каталога, или достаточно только читать уже вычисленные daemon-статусы?
- Нужно ли отображать отдельные визуальные различия между `queued`, `uploading` и `downloading`, или достаточно одного syncing overlay?
- Стоит ли сразу заложить Meson option для пользовательского пути установки `nautilus-python/extensions` для локальной разработки и packaging?
