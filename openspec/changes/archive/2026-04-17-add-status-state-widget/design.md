## Context

Текущее GTK4-приложение состоит из одного окна, собранного прямо в `src/main.vala`, и не содержит выделенных UI-компонентов для отображения состояния. Для демонстрации нового поведения достаточно добавить локальный переиспользуемый виджет и подключить его к существующим обработчикам событий без изменения общей архитектуры приложения.

## Goals / Non-Goals

**Goals:**
- Ввести компактный переиспользуемый виджет состояния для idle, loading и error.
- Сделать переключение состояния простым и явным из кода окна.
- Использовать стандартные GTK4-виджеты без новых зависимостей.
- Показать пример интеграции в основном окне приложения.

**Non-Goals:**
- Не добавлять реальную сетевую загрузку или асинхронный backend.
- Не проектировать универсальную систему уведомлений для всех возможных состояний.
- Не выносить приложение на несколько файлов, если это не потребуется для читаемости.

## Decisions

1. **Выделить отдельный класс `StatusStateWidget` внутри UI-слоя.**  
   Это изолирует представление состояния от логики окна и позволит повторно использовать компонент. Альтернатива — управлять несколькими `Label`/`Spinner` прямо в `MainWindow` — отклонена, потому что она размажет логику состояния по окну и усложнит поддержку.

2. **Использовать enum для явного описания состояний.**  
   Переключение между `IDLE`, `LOADING` и `ERROR` через enum уменьшает риск несогласованных строковых значений. Альтернатива со строковыми ключами проще в быстром прототипе, но хуже масштабируется и хуже проверяется при компиляции.

3. **Построить виджет на контейнере со скрытием/показом внутренних элементов.**  
   Для idle и error достаточно текстового сообщения и иконки/стиля, а для loading нужен `Spinner`. Такой подход хорошо укладывается в GTK4 и не требует дополнительной инфраструктуры. Альтернатива с `Gtk.Stack` тоже возможна, но для трех простых состояний прямое обновление дочерних виджетов дает меньшую сложность.

4. **Интегрировать демонстрационное переключение в существующие обработчики кнопок и полей.**  
   Это позволит показать, как компонент используется в реальном окне, без добавления новых элементов управления только ради демонстрации.

## Risks / Trade-offs

- **[Риск]** Логика состояния останется демонстрационной и не будет отражать реальные асинхронные операции. → **Mitigation:** Явно описать это в design/tasks и оставить API компонента достаточно простым для будущего подключения реальных событий.
- **[Риск]** Размещение нового класса в `src/main.vala` может ухудшить читаемость файла при росте проекта. → **Mitigation:** Держать API виджета компактным; при дальнейшем расширении вынести класс в отдельный файл.
- **[Trade-off]** Отказ от `Gtk.Stack` уменьшает абстракцию состояний, но упрощает реализацию текущего небольшого набора экранов. → **Mitigation:** Сконцентрировать обновление UI в одном методе `set_state`, чтобы последующая миграция на `Gtk.Stack` была локальной.
